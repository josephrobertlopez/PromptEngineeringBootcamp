Prompt Engineering Cookbook

A senior engineer's field guide to systematic AI prompting -- from first principles to production workflows.

1. Three Intuitions

Before any pattern or technique, you need three intuitions. Not facts. Not definitions. Intuitions -- things you feel are true because you've tested them yourself. Everything in this cookbook builds on these three ideas.

Intuition 1: Context Is Everything

The AI doesn't know you. It doesn't know your project, your role, your metrics, your team. Every time you open a chat, it starts from zero. The only thing it works with is what you give it in that moment.

This seems obvious. But most people act as if the AI should "just know" -- and then blame it when the output is generic.

Try It Yourself

Attempt 1 -- No context:

Write my quarterly engineering priority for AI enablement.

Attempt 2 -- Rich context:

I'm a Senior Engineer, 5 years in platform engineering.
My biggest project: data pipeline automation that reduced processing
from 3 minutes to 30 seconds per dataset. I processed 1,500+ datasets
and presented 4 POCs to senior leadership. My AI tool usage includes
daily Copilot and weekly Claude for architecture decisions.

Write my quarterly priority for AI enablement, including:
- 50-100 word description with specific deliverables
- Metrics: AI Tool Usage, Innovation Impact, Knowledge Sharing
- Reflection: Accomplishment, Business Impact, Challenge, Development

Compare the outputs. Same AI. Same model. Same day. The only difference: what you put in.

The intuition: The AI is a context-completion machine. It completes patterns from whatever you give it. Give it nothing, get nothing. Give it your actual work, get actual results. This is the only principle. Everything else is just a different way of getting the right context in front of the AI.

Intuition 2: Structure Gets Rewarded

You might think: "I gave it context, so context is all that matters. Format doesn't matter as long as the info is there." Test that assumption.

Try It Yourself

Take the same information and deliver it two ways:

Wall of text:

I'm a senior engineer with 5 years experience in platform engineering
and my biggest project was data pipeline automation where I reduced
processing from 3 min to 30 sec per dataset and I processed over
1500 datasets and presented 4 POCs to senior leadership and I use
copilot daily and claude weekly for architecture so write my quarterly
priority for ai enablement with description metrics and reflection

Structured:

## Role
Senior Engineer, Platform Engineering (5 years)

## Key Achievement
Data pipeline automation: 3 min -> 30 sec per dataset
- 1,500+ datasets processed
- 4 POCs presented to senior leadership

## AI Tool Usage
- Daily: GitHub Copilot (code assistance)
- Weekly: Claude (architecture decisions)

## Task
Write quarterly priority for AI Enablement:
- Description: 50-100 words, specific deliverables
- Metrics: AI Tool Usage, Innovation Impact, Knowledge Sharing
- Reflection: all four components with evidence
- Format: Ready for submission
The intuition: The AI mirrors the structure you give it. Organized input produces organized output. The model was trained on millions of examples where humans preferred clear, structured responses to clear, structured requests. Headings, bullet points, labeled sections -- these aren't optional formatting. They're signals the AI was trained to respect.

Intuition 3: You Are the Retrieval System

In the structured example above, you typed all the context. You pulled it from your memory, organized it, and pasted it in. You were the retrieval system.

Now imagine: what if a file did that for you? What if a tool did it? What if a platform did it?

Every "AI tool" you'll encounter -- from simple prompts to enterprise platforms -- is doing the same three things:

  1. Retrieving relevant information from somewhere
  2. Assembling it into structured context
  3. Generating output from that context

When you type context manually, you are the retrieval system. When you save it in a file and paste it, the file is the retrieval system. When an enterprise platform pulls from databases, the platform is the retrieval system.

The intuition: Whenever you see an AI tool doing something impressive, ask: "Where is it getting its context from?" The answer is always: somewhere specific. There's no magic. Just good retrieval.

Summary

#IntuitionImplication
1Context is everythingMore relevant context = better output. Always.
2Structure gets rewardedOrganized input produces organized output.
3You are the retrieval system (until you automate it)Every AI interaction is: retrieve -> assemble -> generate.

2. Three Approaches

There are three valid ways to apply prompt engineering patterns. The patterns matter. The format doesn't. Choose based on team maturity and task complexity, not dogma.

The Tier Framework

Before evaluating any approach, use this maturity framework:

TierAgeEvidenceAction
Tier 110+ yearsMicrosoft, AWS, Google, Netflix use itAdopt now
Tier 21-3 yearsGrowing adoption, 1000s of usersAdopt with monitoring
Tier 3MonthsExperimental, limited production dataExperiment cautiously

Questions to ask about any tool: How old is it? Who uses it besides its creator? Does it work cross-platform? What problem does it actually solve?

Overview Comparison

A: FreestyleProvenNoneQuick tasks, exploration
B: SystematicProvenLow-MediumMost teams, repeated tasks
C: Spec-DrivenExperimentalMedium-HighComplex tasks, learning, audit trails

The Same Task, Three Ways

A: Freestyle

Raw chat, no structure. You type what you need.

from Spring Boot 2 to Spring
Boot 3.

When: Simple tasks, brainstorming, exploration.
Risk: Inconsistent quality, no reusability.

B: Systematic

Structured prompts using named patterns + config files.

Following ADR 0001 and
.github/copilot-instructions.md,
migrate UserController.java
to Spring Boot 3.

When: Production teams, medium complexity.
Advantage: Proven, low maintenance, cross-tool.

C: Spec-Driven

Multi-file workflow: knowledge-base -> specification -> implementation plan.

Load spec/knowledge-base.md
Load spec/specification.md
Execute spec/implementation-plan.md
Generate migrated code.

When: Complex multi-phase tasks, learning patterns.
Advantage: Full audit trail, explicit reasoning.

The Spec-Driven Workflow

When tasks are complex enough to warrant it, the spec folder pattern separates concerns into three files:

FileScopeQuestion It Answers
knowledge-base.mdProject-level (reusable)What stays constant? Domain terms, constraints, past decisions.
specification.mdFeature-level (per task)What's unique? Requirements, acceptance criteria, edge cases.
implementation-plan.mdExecution-level (per task)How to execute? Phases, dependencies, validation checkpoints.
The 3-Question Framework: For any domain, ask: (1) What stays constant? (2) What's unique to this instance? (3) How do we execute? These three questions produce the three files. The framework is domain-agnostic -- it works for code migrations, strategic planning, training design, or any structured work.

Decision Flowchart

Is this a simple task?
  Yes -> Freestyle or Systematic prompt
  No  -> Continue

Are you learning patterns or need an audit trail?
  Yes -> Spec-Driven (structured files)
  No  -> Continue

Does your team use multiple AI tools?
  Yes -> Systematic (ADRs + config files)
  No  -> Tool-Assisted workflow in your IDE

3. Foundational Patterns

These four patterns are research-backed (White et al., 2023; Brown et al., 2020; Wei et al., 2022), proven over years of use, and form the foundation of effective prompt engineering. Master these before moving to advanced patterns.

Pattern 1

Persona Pattern

What It Is

Assign the AI a specific role, identity, or expertise level. This sets context, activates relevant knowledge domains, and establishes appropriate tone.

When to Use

  • Complex domains requiring specific expertise
  • Need consistent tone across multiple prompts
  • Want to activate specialized knowledge

Template

You are [role/identity with expertise].

Your responsibilities include [key activities].
Your knowledge covers [domains].

[Task/question]

Example

You are an expert Spring Boot migration specialist
with 10+ years experience upgrading enterprise applications.

Your expertise includes:
- Package migration (javax -> jakarta namespace)
- Spring Security 6.x patterns
- Modern annotation usage

Task: Review this controller and suggest Spring 3 upgrades.

When to Skip

Simple, obvious tasks. General questions not requiring expertise.

Pattern 2

Few-Shot Pattern

What It Is

Provide example input/output pairs to demonstrate desired behavior. Shows rather than tells, establishes format expectations, reduces ambiguity.

When to Use

  • Pattern-based transformations
  • Consistent formatting needed
  • Complex rules that are hard to describe in words

Template

Transform inputs using these examples:

Example 1:
Input: [example input 1]
Output: [example output 1]

Example 2:
Input: [example input 2]
Output: [example output 2]

Now transform:
Input: [actual input]
Output: ?

Example

Transform Java imports using these examples:

Example 1:
Input: import javax.validation.Valid;
Output: import jakarta.validation.Valid;

Example 2:
Input: import javax.servlet.http.HttpServletRequest;
Output: import jakarta.servlet.http.HttpServletRequest;

Now transform:
Input: import javax.persistence.Entity;
Output: ?

When to Skip

One-off unique tasks. Rules that are simple and obvious.

Pattern 3

Template Pattern

What It Is

Provide a structured format for AI output. Ensures consistency, makes responses parseable, easier to validate.

When to Use

  • Need consistent output format across uses
  • Output will be parsed or processed downstream
  • Multiple people using the same prompt

Template

Please provide your response in this format:

## Section 1: [Name]
- Item 1: [description]
- Item 2: [description]

## Section 2: [Name]
| Column A | Column B |
|----------|----------|
| Value    | Value    |

Example

Analyze this code and provide a migration checklist:

## Required Changes
- [ ] Change 1: [description]
- [ ] Change 2: [description]

## Validation Steps
1. [step with expected outcome]

## Success Criteria
[criterion 1]
[criterion 2]

When to Skip

Exploratory questions where format doesn't matter.

Pattern 4

Chain-of-Thought

What It Is

Ask AI to show its reasoning process step-by-step. Improves accuracy on complex reasoning, makes errors debuggable, helps you learn from AI logic.

When to Use

  • Complex debugging or analysis
  • Multi-step reasoning needed
  • Want to understand (and verify) the AI's logic

Template

Solve this problem step by step:

1. First, analyze [aspect 1]
2. Then, consider [aspect 2]
3. Next, evaluate [aspect 3]
4. Finally, conclude [result]

Show your reasoning for each step.

Example

Debug this Kafka consumer issue step by step:

1. First, check the consumer configuration
2. Then, verify the topic exists and has messages
3. Next, examine the consumer group state
4. Then, review offset management
5. Finally, check for serialization issues

Show your reasoning and findings for each step.

When to Skip

Simple factual questions. When you just need the final answer.

Combining Patterns

Patterns compound. Start simple and add layers only when they add value.

You are an expert database performance engineer. (Persona)

Optimize SQL queries using this pattern: (Few-shot)
Before: SELECT * FROM users WHERE active=true
After: SELECT id, name FROM users WHERE active=true

Analyze this query step-by-step: (Chain-of-Thought)
1. Identify performance issues
2. Suggest optimizations
3. Estimate impact

Format your response as: (Template)
## Analysis
[findings]

## Optimizations
- [ ] Change 1
- [ ] Change 2

## Expected Impact
[metrics]

4. Advanced Patterns

These patterns build on the foundational four for complex, multi-step tasks. Use them when simple patterns aren't enough -- but don't force them on simple tasks. Foundational patterns solve 80% of needs.

Advanced

ReAct (Reason + Act)

Source: Yao et al. (2022) -- Multi-step tasks with dependencies and validation checkpoints

What It Is

Structured cycles of reasoning, action, and observation: THINK -> ACT -> OBSERVE, repeated until task complete. Catches errors early, makes dependencies explicit, provides clear rollback points.

Template

## Phase [N]: [Name]

### THINK (Reasoning)
Question: [What must be considered?]
Answer: [Analysis]
Conclusion: [What to do]

### ACT (Action Items)
- Task 1: [specific action]
- Task 2: [specific action]

### OBSERVE (Validation)
Checkpoint: [How to verify]
Success criteria: [What success looks like]
If failure: [What to do]
Dependencies: [What must be complete first]

Example

## Phase 1: Import Updates

### THINK
Question: What dependencies must be satisfied first?
Answer: Imports must compile before annotations can be updated.
Conclusion: Start with javax -> jakarta package updates.

### ACT
- Task 1.1: Replace javax.validation.* with jakarta.validation.*
- Task 1.2: Replace javax.servlet.* with jakarta.servlet.*

### OBSERVE
Checkpoint: Run `mvn compile`
Success criteria: Code compiles without import errors
If failure: Review Spring 3 compatibility docs
Dependencies: None (this is first phase)

## Phase 2: Annotation Modernization
[... continues with same structure ...]

When to Use

Multi-phase migrations. Tasks with clear dependencies. When you need early error detection and clear rollback points.

When to Skip

Single-step tasks. No dependencies between steps. Obvious execution order.

Maps to Production Practices

Phased rollout plans, migration runbooks, CI/CD pipeline stages.

Advanced

Tree of Thoughts

Source: Yao et al. (2023) -- Decisions with multiple valid approaches where tradeoffs matter

What It Is

Explore multiple solution branches, evaluate each, choose best fit: Generate Branches -> Evaluate Each -> Select Best. Prevents premature optimization and documents reasoning for future reference.

Template

## Decision Point: [Question]

### Context
[Background and constraints]

### Branch A: [Option A]
- Pros: [benefits]
- Cons: [drawbacks]
- Context fit: [Excellent/Good/Fair/Poor]
- Effort: [hours]
- Risk: [Low/Medium/High]

### Branch B: [Option B]
[same structure]

### Branch C: [Option C]
[same structure]

### Tree Evaluation
| Branch | Fit | Effort | Risk | Recommendation |
|--------|-----|--------|------|----------------|
| A      | ... | ...    | ...  | RECOMMENDED    |
| B      | ... | ...    | ...  | Consider later |
| C      | ... | ...    | ...  | Future work    |

### Decision: Branch [X]
Rationale: [Why this choice is best for context]

When to Use

Multiple valid technical approaches exist. Tradeoffs depend on specific context. Future developers will ask "why X over Y?"

When to Skip

Only one reasonable approach. Decision is obvious. Don't manufacture fake branches.

Maps to Production Practices

ADR "Alternatives Considered" sections, decision matrices, RFC comparison tables.

Advanced

Meta-Prompting (Orchestration)

What It Is

A prompt that orchestrates other prompts and patterns. Synthesizes context from multiple sources into a unified generation instruction. Creates reproducible workflows.

Template

# [Task]: Complete Pattern Synthesis

## Context Synthesis

**From [Source 1] (Pattern X):**
- Key information extracted
- Constraints/rules applied

**From [Source 2] (Pattern Y):**
- Decisions made
- Approach chosen

**From [Source 3] (Pattern Z):**
- Execution order established
- Validation checkpoints defined

## Generation Instructions

Using all context above:
1. [Step combining patterns]
2. [Step showing orchestration]
3. [Step validating result]

Generate [desired output].

When to Use

Complex tasks with multiple concerns. Combining system prompts, task specs, plans, and decisions. Need reproducible workflow.

When to Skip

Simple tasks. Single pattern sufficient. Once you've internalized the patterns, apply them through simpler formats.

Pattern Selection Guide

Task ComplexityPatterns to UsePatterns to Skip
Simple (5-15 min)Persona + Few-shotReAct, Tree of Thoughts, Meta-prompting
Medium (30-60 min)Persona + Few-shot + Template + CoTMeta-prompting (optional: ReAct)
Complex (hours)All patterns orchestratedNothing -- use everything that adds value

5. The Second Brain Method

Most people have domain knowledge trapped in their heads. When they write tickets, prompts, or specs, they produce vague artifacts because they haven't externalized what they know. The Second Brain method fixes this through structured interrogation: 100 questions across two tiers that extract your tacit knowledge into reusable context.

The Core Idea

A "second brain" is a system for storing knowledge outside your head so you can find it when you need it, connect ideas you'd otherwise forget, and feed it to AI for dramatically better outputs.

The link structure IS the RAG advantage. Tags give you flat retrieval: "show me notes tagged #requirements." Links give you graph traversal: "show me requirements AND everything connected to them." You get what you asked for plus related context you forgot to search for.

Two Tiers

TierNamePain PointDeliverable
1Knowledge ExtractionKnowledge in head leads to vague artifacts20-50 linked notes covering your domain
2Knowledge CompositionPrompting AI from scratch every timePolished context document for AI reuse

Tier 1: The 50 Extraction Questions

Section A: Domain Context (Questions 1-10)
  1. What project/product are you working on right now?
  2. In one sentence, what does it do and who is it for?
  3. What's the business problem it solves?
  4. Who are the main stakeholders and what do they care about?
  5. What's the current state vs. the desired state?
  6. What are the 3 biggest constraints you work within? (time, budget, tech, regulatory, etc.)
  7. What decisions have already been made that you can't change?
  8. What's the history? Why does it look the way it does?
  9. What would a new team member need to understand in their first week?
  10. What do people outside your team consistently misunderstand about your domain?

Notes to create: project-overview, stakeholder-map, constraints, history, common-misconceptions

Section B: Requirements & Standards (Questions 11-20)
  1. What does "done" look like for typical work items?
  2. What's the definition of quality in your context?
  3. What are the non-negotiable requirements? (security, compliance, performance, etc.)
  4. What are the "nice to haves" vs "must haves"?
  5. What approval processes exist and who's involved?
  6. What documentation standards do you follow?
  7. What testing/validation is required before shipping?
  8. What are the common acceptance criteria patterns?
  9. What gets work items rejected or sent back?
  10. What does your QA/review process actually check?

Notes to create: definition-of-done, quality-standards, non-negotiables, approval-process, common-rejection-reasons

Section C: Technical Context (Questions 21-30)
  1. What's the tech stack? (even non-technical people: what tools/systems are involved?)
  2. What integrations or dependencies exist?
  3. What are the known technical constraints or limitations?
  4. What's fragile or risky to change?
  5. What environments exist? (dev, staging, prod, etc.)
  6. What data is involved and where does it live?
  7. What are the common technical gotchas?
  8. What's the deployment/release process?
  9. What monitoring or observability exists?
  10. What technical debt are you carrying?

Notes to create: tech-stack, integrations, fragile-areas, environments, technical-debt

Section D: Patterns & Anti-Patterns (Questions 31-40)
  1. What's a well-written ticket in your context? (give an example)
  2. What's a badly-written ticket? Why did it fail?
  3. What patterns keep recurring in your work?
  4. What mistakes do people make repeatedly?
  5. What shortcuts exist that people should know about?
  6. What "obvious" solutions don't actually work and why?
  7. What tribal knowledge exists that isn't documented?
  8. What questions do new people always ask?
  9. What do you wish someone had told you when you started?
  10. What's the "right way" to do things vs. what actually happens?

Notes to create: good-ticket-example, bad-ticket-example, recurring-patterns, common-mistakes, tribal-knowledge

Section E: People & Process (Questions 41-50)
  1. Who needs to be involved in what types of decisions?
  2. Who has context that others lack?
  3. What communication norms exist? (async, sync, tools, etc.)
  4. What meetings matter and what do they accomplish?
  5. What's the escalation path when things go wrong?
  6. Who are the bottlenecks and why?
  7. What politics or sensitivities should people be aware of?
  8. What's the feedback loop for completed work?
  9. How do priorities get set and changed?
  10. What do you personally know that your team doesn't?

Notes to create: decision-matrix, key-people, communication-norms, escalation-path, hidden-context

Tier 2: The 50 Composition Questions

Section F: Current AI Usage (Questions 51-60)
  1. What AI tools do you currently use?
  2. What do you use them for? (code, writing, analysis, etc.)
  3. What works well? What outputs do you actually use?
  4. What doesn't work? What outputs do you always have to fix?
  5. What context do you repeatedly explain to AI?
  6. What do you copy-paste into prompts frequently?
  7. What prompts do you reuse vs. write fresh?
  8. How much back-and-forth does it take to get useful output?
  9. What would "AI understands my context" look like?
  10. What's the highest-value task AI could help you with if it had full context?

Notes to create: current-ai-workflow, what-works, what-fails, repeated-context, high-value-ai-tasks

Section G: Role & Identity (Questions 61-70)
  1. What's your role? What are you responsible for?
  2. What decisions do you make vs. defer?
  3. What's your expertise? What do you know deeply?
  4. What's your perspective or opinion on how things should be done?
  5. What standards do you hold yourself to?
  6. What tone/style do you communicate in?
  7. What are your pet peeves or non-negotiables?
  8. What do you want AI to assume about you?
  9. What should AI never assume about you?
  10. If AI were your assistant, what would a good one know about you?

Notes to create: my-role, my-expertise, my-standards, communication-style, ai-assumptions

Section H: Output Standards (Questions 71-80)
  1. What does good output look like in your context? (examples)
  2. What's the format/structure you prefer?
  3. What level of detail is right? (terse vs. thorough)
  4. What terminology should AI use or avoid?
  5. What common AI outputs do you always have to fix?
  6. What would make AI output copy-paste ready?
  7. What's the review process for AI-generated content?
  8. What gets rejected and why?
  9. What style guides or standards apply?
  10. How do you measure whether AI output was useful?

Notes to create: good-output-examples, format-preferences, terminology, common-ai-fixes, output-criteria

Section I: Task Patterns (Questions 81-90)
  1. What types of tasks do you do repeatedly?
  2. For each task type: what's the input? What's the expected output?
  3. What context does each task type require?
  4. What are the common variations within each task type?
  5. What's the workflow from request to completion?
  6. What templates or structures do you use?
  7. What checklists or validation steps exist?
  8. What's the 80/20? (20% of tasks that represent 80% of your work)
  9. What tasks could be fully automated vs. need human judgment?
  10. What's the handoff to the next step?

Notes to create: task-catalog, task-templates, task-workflows, high-frequency-tasks, automation-candidates

Section J: Context Composition (Questions 91-100)
  1. Looking at your Tier 1 notes: which are essential for AI to know?
  2. Which notes are "always relevant" vs "sometimes relevant"?
  3. What's the hierarchy? (most important to least important)
  4. What notes should be included by default? What should be pulled in as needed?
  5. What's the right "chunk size" for context? (all at once vs. modular)
  6. How should notes be ordered for maximum comprehension?
  7. What links reveal critical related context?
  8. What's the minimal viable context? (smallest set that still works)
  9. What's the maximal context? (everything that could help)
  10. How do you know when the context document is "done enough"?

Notes to create: essential-context, context-hierarchy, default-inclusions, minimal-context, context-done-criteria

The Deliverable: Your Context Document

After completing both tiers, compose your notes into a reusable context document:

# [Your Name]'s Context File

## Who I Am
[Role, expertise, standards -- from Section G notes]

## My Domain
[Project context, constraints, stakeholders -- from Tier 1 notes]

## How I Work
[Task patterns, workflows, output standards -- from Sections H/I notes]

## What Good Looks Like
[Examples, format preferences, terminology -- from Section H notes]

## AI Instructions
[What to assume, what to avoid, how to communicate -- from Section G notes]

Testing Your Context Document

  1. Take a real task you need to do
  2. Use your context document as preamble / system prompt
  3. Execute the task with AI
  4. Rate: How much correction was needed? What context was missing? What was irrelevant noise?
  5. Update and repeat 3-5 times until the document stabilizes
The before/after test: Do the same task twice -- once with no context document, once with it. The delta demonstrates the value of externalized, composed knowledge. If AI output goes from >50% correction needed to <20%, your context document is working.

Time-Saving Habits

  1. Build a prompt library. Save your best prompts in a folder. Reuse and refine.
  2. Use variables. Write [LANGUAGE], [TASK], [PATTERN] -- replace per use.
  3. Chain prompts. Prompt 1: Analyze. Prompt 2: Using analysis, suggest fixes. Prompt 3: Generate code.
  4. Keep a win log. Track what worked: date, task, pattern used, time saved, what worked.
  5. Start simple. One pattern at a time. Add complexity only when it adds value.
